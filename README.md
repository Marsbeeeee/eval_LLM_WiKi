# eval_LLM_WiKi

面向 LLM 评测与案例复盘的长期知识库。这个仓库采用 Karpathy 在 **LLM Wiki** 中提出的思路：不要让模型每次都从零重新检索、拼接和总结原始材料，而是让 LLM 持续维护一个结构化、可追溯、会累积的 Markdown Wiki。

本项目的目标是把 LLM 评测过程中的原始材料、测试结果、失败案例、修复结论和复盘经验沉淀成一个可持续演进的知识层，方便后续分析模型表现、定位提示词或系统设计问题，并复用已有经验。

## 背景

常见的 LLM 评测资料通常散落在聊天记录、测试 JSON、临时脚本、人工备注和报告里。每次分析新问题时，往往需要重新翻材料、重新喂给模型、重新总结，容易出现：

- 同类 badcase 重复分析，结论不能复用。
- 评测口径、指标、样例说明前后不一致。
- 原始材料和最终结论之间缺少清晰引用链。
- 新增案例没有反哺已有知识库，经验无法累积。

`eval_LLM_WiKi` 希望把这些材料编译成一套长期维护的 Wiki：原始资料保持不可变，LLM 负责更新结构化页面、索引、交叉引用和日志，人负责提供材料、审核结论和决定下一步分析方向。

## 核心思想

本仓库参考 Karpathy 的 LLM Wiki 三层架构：

1. **Raw sources**：原始评测材料，是事实来源，不直接修改。
2. **Wiki**：LLM 维护的 Markdown 知识层，用于总结、归类、关联和沉淀结论。
3. **Schema / Agent instructions**：约束 LLM 如何读取材料、更新页面、记录日志、处理冲突和引用来源。

区别于传统 RAG 的一次性检索回答，这里的 Wiki 是一个会增长的中间产物。每次评测、复盘或问题分析完成后，结论都应该写回 Wiki，让后续问题建立在已有沉淀之上。

## 协作流程

多人共同维护时，建议把人的操作和 Agent 的操作分开：人负责提交材料、说明目标、审核结论；Agent 负责读取材料、更新 Wiki、补充索引和记录日志。Agent 的详细行为规则见 `AGENTS.md`。

1. **提交原始材料**：把评测样本、对话记录、badcase、报告或截图放入 `raw/`，不要直接覆盖历史原始文件。
2. **说明处理目标**：发起任务时明确是 `ingest`、`query` 还是 `lint`，并列出需要处理的 source 路径。
3. **让 Agent 更新 Wiki**：Agent 读取 `wiki/index.md`，查找相关页面，决定新增页面、更新页面或合并结论。
4. **检查引用链**：维护者确认关键结论都能追溯到 `raw/` 或已有 Wiki 页面。
5. **审核不确定项**：对 `待确认` 内容给出人工判断，或补充新的原始材料。
6. **处理无效材料**：误放、重复、被替换或不再使用的 raw 文件统一移动到 `raw/_trash/`，保留 15 天；15 天后确认无引用、无审计价值、无恢复需求，再永久删除。
7. **保留日志**：每次维护都应在 `wiki/log.md` 留下记录，方便追踪知识库如何演进。

推荐给 Agent 的最小任务描述：

```markdown
请按 eval_LLM_WiKi 维护规范处理以下任务：

Task: ingest / query / lint
Source:
- `raw/...`

要求：
- 先读取 `wiki/index.md`。
- 新增或更新相关 wiki 页面。
- 更新 `wiki/index.md` 和 `wiki/log.md`。
- 区分事实、推断和待确认内容。
```

## 推荐目录结构

当前仓库处于初始化阶段，建议逐步按下面结构落地：

```text
eval_LLM_WiKi/
├── README.md                 # 项目入口说明
├── AGENTS.md                 # LLM/Agent 维护 Wiki 的行为规范
├── raw/                      # 原始材料，只读事实来源
│   ├── datasets/             # 评测数据、样本、输入文件
│   ├── conversations/        # 对话记录、badcase trace
│   ├── reports/              # 原始报告、人工记录
│   ├── assets/               # 图片、截图、附件
│   └── _trash/               # 回收站，待删除材料保留 15 天
├── wiki/                     # LLM 生成和维护的知识库
│   ├── index.md              # 内容索引，按主题/实体/任务导航
│   ├── log.md                # 时间线日志，记录 ingest/query/lint
│   ├── evaluations/          # 单次评测复盘
│   ├── badcases/             # badcase 分类与分析
│   ├── prompts/              # 提示词、系统约束、优化记录
│   ├── models/               # 模型表现、版本差异、能力边界
│   └── concepts/             # 通用概念、指标、判断标准
├── tools/                    # 可选脚本，如扫描、检查、导出
└── outputs/                  # 生成结果、临时产物、批处理输出
```

## 工作流

### 1. Ingest：录入新材料

把新的评测样本、对话记录、badcase、报告或截图放入 `raw/`。然后让 LLM 按以下顺序处理：

1. 读取原始材料，确认来源、时间、任务类型和评测对象。
2. 提取关键事实、失败现象、触发条件和人工判断。
3. 在 `wiki/` 中新增或更新相关页面。
4. 更新 `wiki/index.md`，保证后续可以快速定位。
5. 在 `wiki/log.md` 追加一条记录，说明本次处理了什么、改了哪些页面。

### 2. Query：基于 Wiki 提问

分析问题时优先读取 `wiki/index.md`，再打开相关页面，而不是默认重新读取所有原始材料。典型问题包括：

- 某类 badcase 是否重复出现？
- 某个模型版本在哪些任务上退化？
- 某条系统提示词违反了哪些合规约束？
- 本轮评测和上一轮评测的主要差异是什么？
- 哪些结论仍缺少原始证据？

如果一次 Query 产生了可复用结论，应把它写回 `wiki/`，而不是只停留在聊天记录中。

### 3. Lint：健康检查

定期让 LLM 对 Wiki 做一致性检查：

- 是否存在互相矛盾的结论。
- 是否有页面没有被 `index.md` 收录。
- 是否有重要概念被多次提到但没有独立页面。
- 是否有过期结论被新评测覆盖。
- 是否有 badcase 缺少原始材料引用。
- 是否有日志缺失或格式不一致。

## 页面约定

建议 Wiki 页面采用简洁、稳定的 Markdown 结构。示例：

```markdown
# 页面标题

## 摘要

一句话说明这页解决什么问题。

## 关键事实

- 事实 1。来源：`raw/...`
- 事实 2。来源：`raw/...`

## 分析

解释现象、原因、影响范围和与其他页面的关系。

## 相关页面

- [[another-page]]

## 待确认

- 仍缺少哪些证据或人工判断。
```

## 日志格式

`wiki/log.md` 建议使用可搜索的固定前缀：

```markdown
## [2026-06-16] ingest | 文件或主题名称

- Source: `raw/...`
- Updated: `wiki/...`, `wiki/index.md`
- Summary: 本次新增或修正了什么。
```

这样可以用简单命令查看最近操作：

```powershell
Select-String -Path wiki/log.md -Pattern '^## \[' | Select-Object -Last 5
```

## 维护原则

- **原始材料不可变**：`raw/` 是事实来源，修订应通过新增材料或备注完成；如果使用更新版替代旧文件，旧文件先移动到 `raw/_trash/`。
- **不直接删除 raw 文件**：误放、重复、被替换或废弃的 raw 文件统一进入 `raw/_trash/`，保留 15 天；超过 15 天且确认无引用、无审计价值、无恢复需求后，才可永久删除。
- **Wiki 结论可以修正**：如果 digest/ingest 总结错了，直接修正对应 `wiki/` 页面，并在 `wiki/log.md` 记录修正原因。
- **结论必须可追溯**：重要判断需要指向具体原始文件或评测记录。
- **先更新索引，再扩展细节**：`index.md` 是进入 Wiki 的主要地图。
- **保留不确定性**：缺证据时写 `待确认`，不要补造结论。
- **让知识复利**：任何有复用价值的分析，都应该沉淀为页面或更新到已有页面。
