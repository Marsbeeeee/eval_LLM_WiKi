**结论**：

- **多数据集验证通过**：turn-aware、instruction obfuscation、language switch 均跑通，验证可以上线。
- **打分与 deepseek-v3.2 能力相当**：均分 3.99 vs 3.90，低分细节略有差异但属个案（如对 remark 固定话术的过分恪守、tools 拼写错误的处理方式），与模型能力无关。

## 05/07　Judge 模型初步横评（Qwen 3.6 vs DeepSeek 3.2）

**背景**：对比离线 judge 与在线 judge 的评估表现。

**结论**：

- **评分分布高度一致**：两模型在大部分 case 上的打分分布接近，具备基础稳定性。
- **高/低分 case 具备自动筛选价值**：4–5 分与 1–2 分的 case 可给出明确原因，适合用于减少人工初筛工作量。
- **五分制内部区分度不足 → 提出三分制方案**：高分段（4–5）与低分段（1–2）内部无法形成差异辨析。

## 05/09　Mandiri 三分制 human vs model 对比

**背景**：在 Mandiri 数据集上完成三级制人工 vs 模型 judge 对比测试。

**结论**：

- **人机一致性较高（4/5 节点）**：仅 1 处分歧——Gemini 给出 2 分，其余模型与人工均为 3 分，且理由可以接受。
- **三分制有效性验证通过**：将 criteria 区分 should/must 梯度后，多模型打分稳定性显著提升。
- **Qwen 存在模型能力问题**：出现 metrics 与 review 给分不符、review.name 未填入的问题，其他模型均未出现。建议将 metrics 与 review 解耦。（仍然存在问题，qwen 不生成 metrics）

## All time　难点总结

1. **出错溯源困难**：pipeline 运行不提供报错信息，跑通失败难以找到情况，依赖向平台询问根源，效率较为低下
2. **高质量评测数据不足：**当前主要使用的 turn-aware 与 mandiri 数据集区分度有限，不同 judge 模型之间未能稳定拉开显著差异。
3. **评测标准不够稳定统一**：缺少可复用、可标准化的判分规范；面对歧义样本时，人工标注者之间一致性不足，评分方差较大。
4. **模型稳定性受 remark 质量强依赖**：当 remark 数据质量偏低时，pipeline 输出更易被数据噪声主导，难以客观反映模型真实能力。
5. **记录问题意识不足**：遇到难点往往不会优先考虑留档，都是解决完后直接记录任务结论和后续 to do
